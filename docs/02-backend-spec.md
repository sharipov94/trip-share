# Backend Technical Specification

## Цель

Backend обеспечивает:

- авторизацию пользователей через Telegram;
- хранение путешествий;
- управление активностями;
- управление расходами и расчёт баланса;
- уведомления;
- обработку фотографий;
- генерацию Travel Summary.

## Технологии

Рекомендуемый стек:

- **Node.js** + **NestJS** — модульная структура, DI, guards/interceptors.
- **PostgreSQL** — основное хранилище (см. [03-database-schema](03-database-schema.md)).
- **Redis** — кеш, rate-limit, backend для очередей.
- **BullMQ** — фоновые задачи.
- **S3** — хранение фото/чеков/сгенерированных медиа.
- **Docker** — контейнеризация.

## Общие принципы API

- **Ресурс-ориентированность.** Вложенные ресурсы висят на поездке, чтобы контекст
  и проверка доступа были естественными: `POST /trips/:tripId/activities` вместо
  глобального `POST /activities`.
- **Версионирование** через префикс `/api/v1`.
- **Пагинация** для всех списков: курсорная — `?limit=&cursor=`; ответ содержит
  `{ data, nextCursor }`.
- **Аутентификация** — `Authorization: Bearer <JWT>` на всех эндпоинтах, кроме
  `POST /auth/telegram` (см. [07-auth](07-auth.md)).
- **Авторизация** — guard проверяет членство в поездке и роль (см.
  [06-access-control](06-access-control.md)).
- **Идемпотентность** мутаций денег/создания ресурсов — заголовок
  `Idempotency-Key` (см. [05](05-expenses-and-settlements.md#идемпотентность)).
- **Формат ошибок** — единый: `{ error: { code, message, details? } }`.
- **Валидация** входных DTO через `class-validator`.

## Основные модули

### Auth Module
Функции: Telegram Login Validation, выдача JWT, Refresh Token.

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/auth/telegram` | Валидация `initData`, выдача access + refresh |
| POST | `/auth/refresh` | Ротация refresh-токена, новый access |
| POST | `/auth/logout` | Инвалидация refresh-токена |
| GET | `/auth/me` | Текущий пользователь из токена |

### User Module

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/users/me` | Профиль текущего пользователя |
| PATCH | `/users/me` | Обновить профиль, реквизиты, настройки уведомлений |
| GET | `/users/:id` | Публичный профиль участника |

### Trip Module

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/trips` | Создать поездку (создатель → роль `owner`) |
| GET | `/trips` | Список поездок текущего пользователя (пагинация) |
| GET | `/trips/:id` | Детали поездки |
| PATCH | `/trips/:id` | Изменить (owner/admin) |
| DELETE | `/trips/:id` | Удалить (owner) |
| POST | `/trips/:id/invite` | Создать инвайт-токен → deep-link для share |
| POST | `/trips/:id/join` | Присоединиться по инвайту (матчинг по `telegram_id`) |
| DELETE | `/trips/:id/members/:userId` | Удалить участника (owner/admin) |
| GET | `/trips/:id/balance` | **Рассчитанный баланс и план переводов** |
| GET | `/trips/:id/summary` | Travel Summary |

**Добавление людей (только Telegram-контакты).** Mini App **не имеет доступа к списку
контактов** пользователя, поэтому приглашение строится на нативном share Telegram, а не
на серверном матчинге контактов:

1. Клиент вызывает `POST /trips/:id/invite` → бэкенд создаёт запись в `trip_invites`
   (`token`, опционально `expires_at`/`max_uses`) и возвращает deep-link
   (`https://t.me/<bot>/<app>?startapp=<token>`).
2. Клиент открывает нативный Telegram share с этой ссылкой; пользователь выбирает
   контакты сам.
3. Приглашённый открывает ссылку → Mini App читает `start_param` (= `token`) →
   `POST /trips/:id/join`. Пользователь идентифицируется из `initData` по
   `telegram_id`; при первом входе создаётся `users` (регистрация не требуется).

Серверного эндпоинта «матч контактов» в v1 нет — он появится вместе с мобильным
приложением v2 (матчинг по номеру телефона).

### Notification Module
Отвечает за: напоминание за 24 ч, напоминание за 2 ч, photo-запросы, уведомления о

### Summary Module
Генерирует: travel statistics, photo collections, activity summaries, travel story.
Эндпоинт: `GET /trips/:id/summary`.

## Фоновые задачи (BullMQ)

| Очередь | Назначение |
|---------|-----------|
| `reminders` | Напоминания за 24 ч / 2 ч / перед стартом |
| `photo-prompts` | Случайные запросы фото во время активности |
| `receipt-ocr` | Распознавание чеков |
| `summary-generation` | Сбор статистики и Travel Story |
| `debt-reminders` | Мягкие напоминания о невозвращённых долгах |
| `time-capsule` | Отложенное «год назад» напоминание |

Принципы: задачи идемпотентны (повторный запуск не дублирует эффект), с
ограничением попыток и backoff. Планирование напоминаний — через delayed jobs
с `jobId`, привязанным к активности (чтобы при изменении времени можно было
отменить/перепланировать).

**Капсула времени.** При завершении поездки в очередь `time-capsule` ставится
delayed-задача с задержкой ~1 год (`jobId = capsule:{tripId}`). На исполнении задача
выбирает лучший кадр и яркую цифру из `summaries` и шлёт push каждому участнику.
Задержка > TTL обычного брокера — хранить в durable-хранилище (BullMQ поверх Redis
с persistence) или продублировать запись в БД как страховку от потери джоба.

## Группировка фото в «момент»

Для артефакта «один момент — N ракурсов» (см.
[концепт](01-product-concept.md#один-момент--n-ракурсов-ключевой-артефакт)) нужно
объединять фотографии разных авторов, снятые в близкое время в рамках одной
активности.

- Кандидаты — `memories` с одинаковым `activity_id`.
- Кластеризация по `taken_at` методом «временного окна»: фото попадают в один
  момент, если разрыв между соседними по времени кадрами ≤ порога (например, 5 мин).
- Порог конфигурируемый; группировка пересчитывается при добавлении нового фото
  (или фоновой задачей, если объём большой).
- Результат — логические «моменты» (можно хранить как `memory_moment_id` на
  `memories` или вычислять на лету для небольших поездок).
- Приоритет — моменты, собранные по одному `photo-prompt` (у них общий триггер и
  заведомо одно событие).

## Telegram Bot-инициативы

Помимо доставки уведомлений, бот используется как нативная точка входа:

- **Inline-режим** (`@travelmate баланс`): обработчик `inline_query` отвечает
  карточкой с балансом поездки пользователя (данные из `GET /trips/:id/balance`).
  Доступ ограничен участником поездки.
- **Нативные опросы:** при создании активности можно отправить Telegram Poll в
  привязанный групповой чат; обновления голосов приходят через webhook бота и
  маппятся в `activity_votes`.
- **Live-локация (road trip):** подписка на `edited_message` с `location` —
  переход в статус «в пути», накопление трека для километража в статистике.

## Реалтайм

Группа взаимодействует одновременно (голосование, добавление расходов), поэтому
нужен push-канал. Рекомендуется **WebSocket** (NestJS Gateway) с комнатой на
поездку `trip:{id}`; события: `activity.created`, `vote.updated`,
`expense.created`, `balance.updated`, `memory.added`. Fallback — polling
`GET /trips/:id?since=` для слабой сети.

## Хранилище (S3)

Бакеты: `trip-photos`, `receipts`, `summary-assets`, `generated-media`.

Требования к загрузке:
- лимит размера файла и whitelist MIME-типов;
- стрип EXIF-геоданных перед публикацией (privacy);
- доступ через presigned URL или прокси-эндпоинт, не публичный бакет;
- раздача через CDN (CDN URL хранится в БД).

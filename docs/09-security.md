# Security — модель угроз и контроли

Документ фиксирует, как защищается продукт. Контекст рискованный: приложение хранит
**деньги, платёжные реквизиты, банковские скрины и приватные фото с геолокацией**.

Текущая оценка зрелости: аутентификация продумана хорошо; **авторизация и защита
данных — приоритет для доработки до первого реального пользователя с деньгами**.

## Принципы

1. **Deny by default.** Любой доступ запрещён, пока явно не разрешён проверкой
   членства в поездке и роли.
2. **Сервер — источник истины.** Клиенту не доверяем: ни суммам, ни ролям, ни
   `user_id` из тела запроса.
3. **Минимум данных.** Не собираем и не храним лишнего; чувствительное — шифруем и
   удаляем по истечении надобности.
4. **Defense in depth.** Несколько слоёв (валидация → авторизация → шифрование →
   аудит), а не один.

## Поверхности атаки (surfaces)

| Surface | Аутентификация | Главные риски |
|---------|----------------|---------------|
| Mini App → REST API | JWT (из initData) | IDOR, mass assignment, инъекции |
| Telegram Bot (inline, webhook) | **не JWT** — отдельный канал | подделка webhook, утечка баланса через inline |
| S3 (фото, чеки, банк-скрины) | presigned / proxy | публичный доступ к финансовому PII |
| PostgreSQL | сетевой периметр | дамп БД → плейнтекст реквизитов |
| BullMQ / Redis | сетевой периметр | выполнение задач без переавторизации |

## 1. Аутентификация

> Подробности — [07-auth](07-auth.md). Здесь — контроли.

- Валидация `initData`: HMAC-SHA256 (secret = HMAC("WebAppData", BOT_TOKEN)),
  **constant-time** сравнение хэша, проверка `auth_date` TTL (replay-защита).
- `BOT_TOKEN` — только на бэкенде, в secret-manager. Компрометация = полный обход
  аутентификации, поэтому ротация и отсутствие в логах/гите.
- JWT: короткий access (~15 мин), refresh с ротацией и отзывом через Redis.
- Rate-limit на `/auth/*` (per-IP и per-device) — защита от перебора.
- Одноразовый обмен `initData` → токены: тот же `initData` не принимается повторно
  (храним использованный `hash`/nonce на время TTL).

## 2. Авторизация (IDOR) — главный риск

Плоские маршруты (`/expenses/:id`, `/activities/:id`, `/memories/...`,
`/settlements/:id`) уязвимы к перебору UUID. Контроли:

- **`TripMemberGuard` на каждом read/write.** Guard резолвит родительскую поездку
  **из самого ресурса в БД**, а не доверяет пути, и проверяет членство текущего
  пользователя. Нет членства → `403/404` (предпочтительно `404`, чтобы не
  раскрывать существование ресурса).
- **Object-level проверки владения** для мутаций (автор/плательщик) — в сервисе,
  поверх ролей. См. матрицу [06-access-control](06-access-control.md).
- **Запрет на повышение прав:** нельзя сменить свою `role`, `payer_id`, статус
  `settled` напрямую через `PATCH` (см. §3 mass assignment).
- Тест-кейс на каждый ресурс: «не-член поездки получает 404 на чужой объект».

## 3. Защита данных и PII

### payment_details (платёжные реквизиты)
Финансовый PII. Дамп БД не должен раскрывать реквизиты.
- Шифрование на уровне приложения (envelope encryption / KMS-ключ), а не плейнтекст.
- Отдавать только владельцу и (для перевода) получателю долга — не всей группе по
  умолчанию.

### receipts — чеки и банковские скрины (высокая чувствительность)
Бакет `receipts` — это **финансовый PII**, а не «картинки». Банковский скрин содержит
номер карты/счёта, остаток, ФИО.
- **Подписанные URL с TTL** или proxy-эндпоинт; бакет приватный (никаких публичных
  ACL).
- Доступ — только участникам поездки (та же object-level авторизация).
- **Маскирование чувствительных полей** при OCR/превью (номер карты → `**** 1234`),
  чтобы группа не видела полные реквизиты плательщика.
- **Ретеншн:** оригинал скрина нужен до подтверждения долга; после — удалять
  оригинал, оставляя только распознанную сумму.

### Данные здоровья
Особая категория. Строгий opt-in, шифрование at-rest, минимальное хранение, лёгкое
удаление.

## 4. Фотографии и геолокация

Фото поездки раскрывают, где человек живёт и когда его нет дома.
- **Стрип EXIF-геоданных** перед публикацией.
- `photo_url` — **не публичная** ссылка: подписанный URL с TTL или proxy. Random key
  в S3 — недостаточная защита («security by obscurity»).
- Доступ к фото — только участникам поездки.

## 5. Telegram Bot и webhook (каналы в обход JWT)

- **Inline `@travelmate баланс`:** запрос приходит без JWT. Маппим
  `inline_query.from.id` → `users.telegram_id` → проверяем членство в поездке.
  Не член — не отдаём баланс.
- **Webhook (опросы, live-локация):** проверяем заголовок
  `X-Telegram-Bot-Api-Secret-Token` (секрет, заданный при `setWebhook`). Запросы без
  валидного секрета отбрасываем — иначе подделка голосов и «мы выехали».
- Бот не отправляет в push чувствительные суммы не тому адресату.

## 6. Валидация ввода и загрузки

- DTO-валидация через `class-validator`; **whitelist полей** (`forbidNonWhitelisted`)
  — защита от **mass assignment** (нельзя протащить `role`, `settled`, чужой `user_id`).
- Параметризованные запросы / ORM — против SQL-инъекций. Никакой конкатенации SQL.
- Файлы: лимит размера, whitelist MIME, **проверка по magic bytes** (не по
  заголовку), **ре-энкод изображений** (против image-bomb и polyglot-файлов).
- S3-ключи — случайные; имя файла из клиента не используется как путь.

## 7. Целостность денег

- **Идемпотентность:** `Idempotency-Key` защищает от **повторной отправки одним
  плательщиком** (двойной тап, ретрай на плохой сети, повтор оффлайн-мутации) — не от
  «двух пользователей». См. [05](05-expenses-and-settlements.md#идемпотентность).
- `CHECK (amount >= 0)`; суммы и доли валидируются на сервере, не берутся «на веру».
- Кто может отметить долг `settled` — строго по [06](06-access-control.md), не любой
  участник.
- Транзакции БД при создании расхода + пересчёте долей (атомарность).
- **Наличные без подтверждения** — не уязвимость, но плательщик вводит непроверяемую
  сумму; на продуктовом уровне желателен механизм подтверждения/оспаривания расхода
  участниками (вынесено в backlog, не в безопасность).

## 8. Инфраструктура и эксплуатация

- **HTTPS** везде; HSTS.
- **CORS allowlist** — только домен Telegram WebApp; security-заголовки и CSP для
  веб-части.
- **Rate-limiting** per-user и per-IP (Redis) на auth, загрузки, инвайты.
- **Секреты** — в secret-manager, не в `.env` на проде, не в гите.
- **Сканирование зависимостей** (Dependabot / `npm audit`) и Docker-образов.
- **Redis/Postgres** не торчат в интернет; доступ по приватной сети.
- Фоновые задачи (BullMQ) при выполнении **перепроверяют актуальность данных и права**
  — состояние могло измениться с момента постановки джоба.

## 9. Логирование и аудит

- В логи **не попадают** секреты, токены, `payment_details`, содержимое чеков.
- **Аудит-лог денежных действий**: кто создал/изменил/удалил расход, кто отметил долг
  оплаченным (для разбора споров).
- Алёрты на аномалии: всплеск 403, перебор UUID, массовые загрузки.

## 10. Приватность / GDPR

Аудитория включает ЕС (пример «Барселона»). Минимум:
- **Право на удаление** — каскадное удаление пользователя и его данных (FK CASCADE
  уже в [схеме](03-database-schema.md)).
- **Экспорт данных** по запросу.
- **Согласие** на данные здоровья и на обработку фото.

## Приоритеты (must-have до первого пользователя с деньгами)

1. Object-level авторизация (IDOR) на всех ресурсах — §2.
2. Шифрование `payment_details` и приватный доступ к `receipts`/банк-скринам — §3.
3. Подписанные URL для фото + EXIF-стрип — §4.
4. Проверка webhook-секрета и членства для inline-бота — §5.
5. Whitelist полей на всех `PATCH` — §6.

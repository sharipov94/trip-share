# Expenses & Settlements — модель денег

> Это ядро продукта и самое рискованное место v1. Документ фиксирует правила,
> которые в исходных спеках были описаны одной строкой («система рассчитывает»).

## Принципы

1. **Источник истины — расходы (`expenses`) и их доли (`expense_participants`).**
   Баланс и план переводов — всегда производные, пересчитываются, а не редактируются
   вручную.
2. **Каждая доля знает свой статус оплаты** (`pending` / `settled`).
3. **Все операции с деньгами идемпотентны.**

## Валюты

- У поездки есть `base_currency` (ISO 4217), заданная при создании.
- Расход хранит свою `currency` и `exchange_rate` — курс к `base_currency` на дату
  расхода. Курс фиксируется в момент создания (snapshot), чтобы прошлые расходы не
  «плавали» при изменении курса.
- Баланс и план переводов считаются **только в `base_currency`**. В UI доля может
  показываться в обеих валютах, но сведение долгов — в базовой.
- Источник курсов — внешний провайдер (например, ECB/openexchangerates),
  кешируется в Redis на сутки. Если курс недоступен — пользователю предлагается
  ввести вручную; `exchange_rate = 1` запрещён для валюты, отличной от базовой.

## Округление

- Хранение — `NUMERIC(14,2)` (2 знака; для валют без копеек — округление до целого
  на уровне представления).
- Деление суммы на N участников **не может терять или создавать деньги**. Алгоритм
  «largest remainder»:

  1. `base = floor(amount / N)` на каждого (в минимальных единицах, копейках);
  2. `remainder = amount − base * N` копеек;
  3. распределить `remainder` по +1 копейке первым `remainder` участникам
     (порядок детерминированный — по `user_id`).

  Пример: 10.00 / 3 → 3.34 + 3.33 + 3.33 = 10.00. Сумма долей всегда равна сумме
  расхода (инвариант проверяется в тесте и `CHECK` на уровне сервиса).

## Режимы разделения (`split_mode`)

| Режим | Когда | Кто участвует |
|-------|-------|---------------|
| `equal` | по умолчанию | все участники поездки |
| `passengers_only` | автомобильный режим (бензин/парковка/платная дорога) | только отмеченные пассажиры |
| `manual` | ручное распределение | заданные доли/суммы |

Для `manual` сумма заданных долей обязана равняться `amount` (валидация на входе).

## Сценарии создания расхода

### Простое разделение
По умолчанию 50/50 (равно) между всеми участниками поездки.

### Активность с известной стоимостью
Стоимость известна заранее → создаётся `expense` с `activity_id`, доли
распределяются по участникам активности, участникам предлагается перевести деньги
плательщику (организатору).

### Активность с неизвестной стоимостью
После завершения (`POST /activities/:id/complete`) организатор вводит итоговую
сумму → создаётся `expense`, создаются доли (`expense_participants`).

### Ресторанный чек
OCR строит список позиций (`receipt_items`); каждый участник отмечает свои позиции
и общие; после подтверждения суммы позиций конвертируются в `expense_participants`
с режимом `manual`.

## Расчёт баланса

Чистая позиция участника:

```
net(u) = Σ(оплачено u как payer) − Σ(доли u как participant, pending)
```

- `net(u) > 0` — участнику должны;
- `net(u) < 0` — участник должен.

Сумма всех `net` по поездке = 0 (инвариант).

## Минимизация переводов

Задача: имея вектор `net` по участникам, построить минимальный набор переводов,
обнуляющий все позиции (debt simplification).

**Алгоритм (greedy, практичный для групп 3–10 человек):**

1. Разделить участников на должников (`net < 0`) и кредиторов (`net > 0`).
2. Положить обе группы в очереди с приоритетом по абсолютной величине.
3. Пока есть должники и кредиторы:
   - взять крупнейшего должника `d` и крупнейшего кредитора `c`;
   - `t = min(|net(d)|, net(c))`;
   - создать перевод `d → c` на сумму `t`;
   - уменьшить обе позиции на `t`, вернуть ненулевые остатки в очереди.
4. Результат записать в таблицу `settlements` (перезаписывается при каждом
   пересчёте незакрытых долгов).

Greedy даёт ≤ N−1 переводов, чего достаточно для целевых групп. (Оптимальная
минимизация — NP-hard в общем случае; для 3–10 человек разница незначима.)

Пересчёт `settlements` запускается при: создании/изменении/удалении расхода и при
отметке доли оплаченной. Уже `settled`-переводы из расчёта исключаются.

## Отметка оплаты

`POST /settlements/:id/settle` помечает перевод как `settled` и соответствующие
`expense_participants` — `settled` + `settled_at = now()`. Действие идемпотентно
(повторный вызов не меняет состояние).

## Закрытие долга в один тап

Цель — свести фактический перевод денег к одному нажатию (см.
[концепт](01-product-concept.md#закрытие-долга-в-один-тап)).

- Рядом с каждым `pending`-переводом UI показывает кнопку «Перевести».
- Кнопка ведёт по **диплинку**: открывает личный чат должника/получателя в Telegram
  с заранее подставленным текстом (сумма, валюта, реквизиты получателя из
  `users.payment_details`). На платформах с платёжными диплинками — прямой переход к
  оплате.
- После перевода инициатор подтверждает оплату → `POST /settlements/:id/settle`
  (идемпотентно). Подтверждение может требовать второй стороны — политика из
  [06-access-control](06-access-control.md).
- Реквизиты получателя берутся из его профиля; если их нет — UI просит получателя
  сначала заполнить `payment_details`.

> Минимум для запуска: диплинк в чат с текстом и реквизитами. Интеграция с платёжными
> провайдерами — позже, поверх той же кнопки.

## Идемпотентность

Расход создаёт **один человек — плательщик** (прикладывая чек или банковский скрин;
для наличных — можно без подтверждения). Сценария «двое заводят один и тот же расход»
нет. Реальный источник дублей — **повторная отправка тем же плательщиком**: двойной
тап, ретрай на плохой сети, повтор отложенной оффлайн-мутации. От этого защищает
`Idempotency-Key` (UUID, генерируется клиентом на действие, а не на запрос):

- ключ сохраняется (`expenses.idempotency_key`, UNIQUE на `(payer_id, key)`);
- повторный запрос с тем же ключом возвращает **существующий** ресурс, а не создаёт
  новый;
- работает в связке с отложенными оффлайн-мутациями
  (см. [04-frontend-spec](04-frontend-spec.md#offline-support)).

> Чек/банковский скрин — финансовый PII (номер карты, остаток). Требования к их
> хранению — в [09-security](09-security.md#receipts--чеки-и-банковские-скрины-высокая-чувствительность).

## Инварианты (проверять тестами)

- Σ долей расхода == `amount` расхода.
- Σ `net` по поездке == 0.
- Число переводов в `settlements` ≤ (число участников − 1).
- Перерасчёт не трогает уже `settled` обязательства.
